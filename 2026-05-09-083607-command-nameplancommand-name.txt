     PROMETHEUS — Workout Templates, Library & Calendar Planning

     Context

     Today, PROMETHEUS persists workouts as prometheus_sessions rows whose label
      doubles as an ad-hoc "template name". Reusing a workout means cloning the
     most recent session that happens to share the same label
     (task_service._auto_create_session_from_label). The UX is rough: the
     workout name is auto-derived ("Bench press + Squat +3"), the only way to
     schedule a workout into the future is window.prompt for a date in
     WorkoutLog, and there is no first-class concept of a saved/reusable
     workout. The user's mental model — I finish a workout, save it under a 
     name, reuse it later, plan it on the calendar — is not represented in the
     schema.

     This plan introduces workout templates as a first-class entity, with a
     library UI, inline calendar planning, "load saved workout for today", and a
      required name on save. Per-exercise kg/reps continue to come from the last
      performance of that exercise (existing getLastSets endpoint), so
     progressive overload still works automatically.

     Decisions (confirmed with user):
     1. "Save" creates both a session record (history) and upserts a template by
      name.
     2. Templates store exercise order + target set count only — kg/reps come
     from last performance at load time.
     3. Calendar planning happens via an inline date picker in the library.
     4. daily_tasks gets a new workout_template_id UUID FK (label column kept
     for back-compat).

     Backend changes

     1. New migration — migrations/016_workout_templates.sql

     - workout_templates (id UUID PK, user_id FK, name TEXT, created_at,
     updated_at); UNIQUE (user_id, lower(name)).
     - workout_template_exercises (id UUID PK, template_id FK ON DELETE CASCADE,
      exercise_name TEXT, order_index INT, target_sets INT DEFAULT 3,
     muscle_load JSONB).
     - ALTER TABLE daily_tasks ADD COLUMN workout_template_id UUID REFERENCES 
     workout_templates(id) ON DELETE SET NULL.
     - RLS enabled on both new tables, user_id = auth.uid() policy per ADR-011.
     - Indexes: (user_id) on templates, (template_id, order_index) on template
     exercises, (workout_template_id) on daily_tasks.

     2. Models — backend/models/schemas.py

     - Add WorkoutTemplateExerciseInput, WorkoutTemplateCreate,
     WorkoutTemplateUpdate, WorkoutTemplate, WorkoutTemplateExercise Pydantic
     models.
     - SessionCreate (already exists at ~line 248): the existing label: str
     field becomes the template name. Add an optional save_as_template: bool = 
     True so ad-hoc sessions can opt out.

     3. New service — backend/services/workout_template_service.py

     - list_templates(supabase) -> list[WorkoutTemplate] — with eager-loaded
     exercises.
     - get_template(supabase, id) -> WorkoutTemplate.
     - upsert_template(supabase, name, exercises) -> WorkoutTemplate — find by
     (user_id, lower(name)); if exists, replace exercises and bump updated_at;
     else create.
     - update_template(supabase, id, payload) — supports rename, reorder,
     add/remove/change target_sets.
     - delete_template(supabase, id).
     - schedule_template(supabase, template_id, dates: list[date]) — for each
     date, upserts a daily_tasks row with task_type='workout',
     workout_template_id=..., workout_template_label=name, title=name,
     category='health', estimated_minutes=60.

     4. Modify backend/services/prometheus_service.py

     - create_session() (lines 125–247): after the session insert succeeds and
     before kcal analysis, call workout_template_service.upsert_template(...)
     when payload.save_as_template is true and payload.label is non-empty. Build
      template exercises from [{exercise_name, order_index, 
     target_sets=len(sets), muscle_load}]. Failures here must not break session
     creation — wrap in try/except, log only.

     5. Modify backend/services/task_service.py _auto_create_session_from_label 
     (lines 308–394)

     - New resolution order: if the daily_task has workout_template_id set,
     fetch exercises from workout_template_exercises and for each call
     prometheus_service.get_last_sets_for_exercise(name) to populate sets (fall
     back to empty sets if exercise was never performed). Otherwise fall back to
      existing label-based session lookup.
     - Keep silent-failure semantics.

     6. New router — backend/routers/workout_templates.py

     Routes (mounted under /api/v1/workout-templates in backend/main.py):
     - GET / → list
     - GET /{id} → detail with exercises + last_sets per exercise (one DB
     roundtrip per exercise via existing get_last_sets_for_exercise)
     - POST / → create
     - PATCH /{id} → update (rename / reorder / target_sets)
     - DELETE /{id}
     - POST /{id}/schedule body { dates: ["YYYY-MM-DD", ...] } → bulk-create
     daily_tasks
     - POST /{id}/start-today → upsert today's daily_task pointing at this
     template; returns the task
